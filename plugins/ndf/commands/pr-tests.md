# PR Test Plan実行

このコマンドは、`/ndf:pr`コマンドで作成されたPRのTest Planを自動実行します。

## 概要

PRの本文に記載されたTest Plan（チェックリスト形式）を読み取り、各テスト項目を自動で実行します。テスト結果に基づいてPRコメントを更新し、成功したテストにチェックマークを付け、失敗したテストについてはコードを修正またはコメントを追加します。

## 引数

- **PR番号**（オプション）: 指定しない場合は現在のブランチから自動検出

**使用例**:
```bash
# 現在のブランチのPRをテスト
/ndf:pr-tests

# 特定のPR番号を指定
/ndf:pr-tests 123
```

## セキュリティ対策

このコマンドは自動コード修正機能を含むため、以下のセキュリティ対策を実装します：

### 必須の安全措置

1. **ユーザー確認プロンプト**
   - コード修正を行う前に必ずユーザーに確認を求める
   - AskUserQuestionツールを使用して明示的な承認を得る
   - 修正内容の概要を事前に提示

2. **変更範囲の制限**
   - Test Plan実行に直接関連するファイルのみを修正対象とする
   - 設定ファイル、認証情報、セキュリティ関連ファイルは自動修正禁止
   - 修正前にファイルの種類を検証

3. **変更内容の検証**
   - 修正前後のdiffを明示的に表示
   - セキュリティスキャン（secret-scan、OWASP基本チェック）を実施
   - 構文エラーがないことを確認

4. **ロールバック機能**
   - 修正に失敗した場合は自動的にgit resetを実行
   - 修正前の状態を保存（git stash活用）
   - エラー時のリカバリー手順を明示

5. **監査ログ**
   - すべての自動修正をPRコメントに記録
   - 修正したファイル、理由、コミットハッシュを明記
   - 修正内容のレビュー依頼を明示

### 禁止事項

以下のファイル・操作は自動修正の対象外とし、必ず手動確認を要求します：

- 認証情報・シークレット（.env、credentials.json等）
- セキュリティ設定ファイル（.gitignore、security.yml等）
- CI/CD設定ファイル（.github/workflows/等）
- データベースマイグレーションスクリプト
- 本番環境設定ファイル
- パッケージマネージャーの依存関係ファイル（package.json、requirements.txt等）

## 実行手順

### 1. PR情報取得

**ステップ**:
- 引数でPR番号が指定されている場合: その番号を使用
- 指定されていない場合: `gh pr view --json number`で現在のブランチのPRを取得
- GitHub MCPまたは`gh pr view`でPRの本文を取得
- Test Planセクションを抽出
- チェックリスト項目をパース（`- [ ]`形式）

**エラーハンドリング**:
- PRが存在しない場合: エラーメッセージを表示して終了
- Test Planが見つからない場合: 警告を表示して終了

### 2. テスト実行計画の立案

**ステップ**:
- 各テスト項目を解析
- テストの種類を判定（機能テスト、統合テスト、手動確認など）
- 実行順序を決定（依存関係を考慮）
- 自動実行可能なテストと手動確認が必要なテストを分類

**テスト種類の例**:
- プラグインのインストール/再読み込み
- コマンドの実行
- ファイルの存在確認
- コードの構文チェック
- セキュリティスキャン
- 機能の動作確認

### 3. テスト実行

**ステップ**:
- 各テスト項目を順次実行
- 実行状況をリアルタイムで報告
- 各テストの成功/失敗を記録
- エラーが発生した場合は詳細を記録

**実行方法**:
- コマンド実行系: Bashツールで実行
- ファイル確認系: Read/Globツールで確認
- コード検証系: 適切なMCPツールを使用
- 手動確認系: ユーザーに確認を求める

### 4. 結果反映

**成功したテストの場合**:
- PRの本文を更新
- `- [ ]`を`- [x]`に変更
- GitHub MCPまたは`gh pr edit`を使用

**失敗したテストの場合**:
- **原因がコードの場合（自動修正フロー）**:

  **⚠️ セキュリティチェックポイント**:
  1. **修正対象ファイルの検証**
     - 禁止ファイルリストと照合
     - セキュリティ関連ファイルでないことを確認

  2. **ユーザー確認プロンプト**
     - AskUserQuestionツールで修正承認を求める
     - 修正内容の概要を提示
     - 「承認する」「スキップする」「中断する」の選択肢を提供

  3. **修正前の状態保存**
     - `git stash push -m "pre-auto-fix-backup"`で現在の状態を保存

  4. **コード修正実施**
     - 承認された場合のみ修正を実行
     - 修正内容をdiffで確認

  5. **修正後の検証**
     - セキュリティスキャン（secret-scan相当）を実施
     - 構文エラーチェック
     - 問題が検出された場合はgit stash popでロールバック

  6. **コミット・プッシュ**
     - `git add` → `git commit` → `git push`
     - コミットメッセージに「[auto-fix] テスト項目名」を含める

  7. **監査ログ記録**
     - PRにコメントを追加（修正内容、理由、コミットハッシュ）
     - GitHub MCPまたは`gh pr comment`を使用

  **エラーハンドリング**:
  - 修正失敗時: `git stash pop`でロールバック
  - エラー詳細をPRコメントに記録
  - 手動修正が必要な旨を明示

- **コードで修正できない場合**:
  1. PRにコメントを追加
  2. 失敗理由と手動確認が必要な旨を説明
  3. 次のアクションを提案
  4. 必要に応じて関連ドキュメントへのリンクを提供

### 5. 最終報告

**テスト結果サマリーをPRにコメント**:

```markdown
## 🧪 Test Plan実行結果

### 実行サマリー

| 項目 | 値 |
|------|-----|
| 総テスト数 | N件 |
| ✅ 成功 | X件 |
| ❌ 失敗 | Y件 |
| ⏭️ スキップ | Z件 |
| 成功率 | XX% |

### テスト結果詳細

#### ✅ 成功したテスト (X件)
1. [テスト項目1]
2. [テスト項目2]

#### ❌ 失敗したテスト (Y件)
1. **[テスト項目3]**
   - エラー: [エラー内容]
   - 対応: [コード修正 / 手動確認必要]
   - コミット: `abc1234`（修正した場合）

#### ⏭️ スキップしたテスト (Z件)
1. [テスト項目4] - 理由: [スキップ理由]

### 次のアクション

- [ ] 失敗したテストの手動確認
- [ ] 修正内容のレビュー
- [ ] 追加修正が必要な場合は対応
```

## 注意事項

### セキュリティ（重要）

**自動コード修正に関する制約**:
- ⚠️ **すべての自動修正は事前にユーザー承認が必要**
- ⚠️ **禁止ファイルリストに該当するファイルは絶対に自動修正しない**
- ⚠️ **修正前に必ずセキュリティスキャンを実施**
- ⚠️ **修正に失敗した場合は自動的にロールバック**

**コミット時の注意**:
- 機密情報（トークン、パスワード等）を絶対にコミットしない
- git操作は安全に実行（mainブランチへの直接プッシュ禁止）
- コミットメッセージに[auto-fix]プレフィックスを付与して自動修正であることを明示

**監査とトレーサビリティ**:
- すべての自動修正をPRコメントに記録
- 修正したファイル、理由、コミットハッシュを明記
- ユーザーが後から修正内容を追跡できるようにする

### エラーハンドリング
- 各テストで発生したエラーは詳細にログ記録
- テスト実行中のエラーで全体を停止しない（可能な限り継続）
- 致命的なエラーの場合は適切にユーザーに報告
- 自動修正に失敗した場合はgit stashでロールバックし、エラー内容を記録

### GitHub API制限
- GitHub APIのレート制限を考慮
- 大量のコメントを一度に投稿しない
- 可能な限りバッチ処理を使用

- **PR読み込み**: `gh pr view [番号] --json body` または GitHub MCP
- **PRコメント追加**: `gh pr comment [番号] --body "コメント"` または GitHub MCP
- **PR本文更新**: `gh pr edit [番号] --body "新しい本文"` または GitHub MCP
- **git操作**: `/ndf:pr`コマンドのロジックを参考

## Test Plan解析のロジック

**パース処理**:
```
1. PR本文を取得
2. "## Test plan" または "## Test Plan" セクションを検索
3. セクション内の "- [ ]" 行を抽出
4. 各行をテスト項目として配列化
5. チェック済み項目 "- [x]" は除外（既に実行済み）
```

**テスト項目の分類**:
- "インストール", "install" → プラグインインストールテスト
- "実行", "run", "動作" → 機能実行テスト
- "ファイル", "file", "存在" → ファイル確認テスト
- "セキュリティ", "security" → セキュリティスキャン
- "確認", "check" → 手動確認（ユーザーに確認を求める）
